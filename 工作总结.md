#                                工作总结

## 一：关于项目流程

```
关于项目流程，我将它归为4个步骤。1.拿到项目需求时的检查，这个环节很重要，可以避免许多问题，提高开发效率。2.代码构造。3.测试环节。4.提交代码至仓库，上线。以下我把这四个步骤细化。
1.需求检查注意事项
  1.先看看原型图是否有错别字，
         联系方式是否正确，
         确认所用的链接是否正确，
         是否需要自适应，手机端跟pc端，
         是否需要兼容各个浏览器，（尤其是ie,uc）
         语句是否通顺。
  2.看需求符不符合实现的标准，先想一下怎么实现最好，不能着急动手。
  3.业务流程、功能是否符合用户体验。
  4.对于一些模棱两可的功能模块要跟需求方确认再三，减少代码功能的更改。
2.代码构造
  1.整体架构，想好布局。
  2.提高代码可重用性，代码整洁。
  3.命名规范化，统一风格。
  4.图片压缩，再看看是否有多余的无用的图片，记得删除，以免影响加载速度。
  5.如果是再之前做过的网页中稍加修改变成另外一个网页的时候，要记得换logo,title,联系方式，链接，并且打开项目的时候不能同时打开多个类似的，容易搞混。修改一次记得提交代码，本地不要存同一个文件多份，容易混淆。只要一个就好。
  6.每天记得提交代码，不一样的项目做完记得提交。
  7.后台代码要记得更新，不能上传旧版本。
  8.删除alert,console,及其他一些无用的注释。
 3.测试环节
  1.测试之前写个测试文档，需要测试的功能块，按流程走完。不能缺胳膊少腿，做好了一个测试好了，又影响了另外一个已经测试好的功能，然后还没有发现。测试要从头至尾测，有bug先记录在一旁。然后统一解决完，再重新走一遍流程。
  2.pc端，手机端各个型号都要测试一下。
 4.提交
   1.提交前看看文件名看看是否提交正确，文件夹是否多上传，是否有多余的无用的文件，包，图片等等。
   2.提交完代码再上传至自己的远程服务器上看看效果。重新走一遍流程。
   3.确认无误之后，再交给需求方测试一遍，若需求方要改动，先让其把所有的要改的地方归纳好，再做修改，然后修改完之后再重新走一遍测试环节及提交环节。
   4.需求方确认无误后再上传至仓库。
   5.交的时候要交代清楚哪些需要替换，哪些不需要替换，以免把自己还没改完的其他代码全部替换掉了。
```
